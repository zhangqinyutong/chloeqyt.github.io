## 问题 1：执行大表操作时，C 盘空间持续爆满、不断缩水  

原因:  
1、MySQL 默认临时目录、数据目录、日志目录全部在 C 盘；  
2、执行 ALTER 加字段 / 全表UPDATE / DELETE 去重 等批量操作时：
系统临时文件、事务 undo/redo 日志、binlog 二进制日志疯狂膨胀；

解决方法：  
执行 SQL 清理二进制日志，快速释放 C 盘空间
```sql
PURGE BINARY LOGS BEFORE NOW();
```
根治方案：迁移 MySQL 数据目录 + 临时目录到 E 盘，后续所有数据、临时文件、日志全部走 E 盘，不再占用 C 盘。

## 数据集截取、新增字段、去异常
1亿条数据运行起来太慢了，缩小数据集。  
MySQL 按天分层抽样，每天抽取约 8% 的数据，总数据量约 800 万条
```sql
-- 复制原表结构
CREATE TABLE user_behavior_sampled LIKE user_behavior;

-- 插入抽样数据（约8%比例，总数据量≈800万条）
INSERT INTO user_behavior_sampled
SELECT * FROM user_behavior
WHERE 
    MOD(CONV(SUBSTRING(MD5(user_id), 1, 8), 16, 10), 100) < 8
    AND datetimes BETWEEN '2017-11-25 00:00:00' AND '2017-12-03 23:59:59';
--MD5(user_id)：对 user_id 做 MD5 哈希运算，生成一个32 位的 16 进制随机字符串。
--SUBSTRING(MD5(user_id), 1, 8)：截取哈希片段
--CONV(..., 16, 10)：16 进制转 10 进制整数
--MOD(..., 100)：把转换后的 10 进制整数对 100 取模，得到一个 0-99 之间的随机整数。
--< 8：只保留取模结果小于 8 的记录（即 0-7，共 8 个数字）


-- 1. 检查每天的数据量是否均衡
SELECT 
    DATE(datetimes) AS 日期,
    COUNT(*) AS 数据量
FROM user_behavior_sampled
GROUP BY DATE(datetimes)
ORDER BY 日期;

-- 2. 检查行为类型分布是否正常（pv≈89%，cart≈6%，fav≈3%，buy≈2%）
SELECT 
    behavior_type AS 行为类型,
    COUNT(*) AS 数量,
    ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM user_behavior_sampled), 2) AS 占比
FROM user_behavior_sampled
GROUP BY behavior_type;
```
(耗时21min)

## 获客情况
- 电商数据分析的基础测试流程
1.先创建一个小数据量的临时表快速验证逻辑  
2.再分别计算每日的 PV（页面浏览量）、UV（独立访客数）、浏览深度（PV/UV） 这三个核心指标

```sql
-- 1. 创建临时表：
CREATE TABLE temp_behavior LIKE user_behavior_sampled;

-- 2. 截取数据：从抽样表取前 10 万条插入临时表
INSERT INTO temp_behavior
SELECT * FROM user_behavior_sampled LIMIT 100000;

```
- 10 万条数据跑 SQL 只需几秒，能快速验证统计逻辑是否正确，避免直接在 800 万条的大表上反复测试浪费时间。逻辑验证通过后，把 temp_behavior 替换成 user_behavior_sampled 就能跑全量数据。
```sql
-- pv：PageView，页面浏览量（用户每打开一次页面就算1次，同一用户多次打开算多次）
SELECT dates,COUNT(*)'pv'
FROM temp_behavior
WHERE behavior_type='pv'  -- 只统计“浏览”行为
GROUP BY dates;           -- 按日期分组，得到每天的总浏览量
```
- 业务含义：PV 反映网站的 “流量规模”，数值越高说明网站越热门。


```sql
-- uv: Unique Visitor，独立访客数（同一天内同一个用户只算1次，不管浏览多少次）
SELECT dates,COUNT(DISTINCT user_id)'uv' 
--COUNT(DISTINCT user_id)：对 user_id 去重后再计数，避免同一个用户被重复统计
FROM temp_behavior
WHERE behavior_type='pv'
GROUP BY dates;
```
- 业务含义：UV 反映网站的 “用户规模”，比 PV 更能体现真实的活跃用户数。

```sql
-- 浏览深度 PV/UV：平均每个用户每天浏览的页面数
SELECT dates
, COUNT(*)'pv'                    -- 每日总浏览量
, COUNT(DISTINCT user_id)'uv'     -- 每日独立访客数
, ROUND(COUNT(*)/COUNT(DISTINCT user_id),1) 'pv/uv'  -- PV除以UV，保留1位小数
FROM temp_behavior
WHERE behavior_type='pv'
GROUP BY dates;
```
- 业务含义：
浏览深度越高，说明用户对网站内容越感兴趣，愿意花时间逛；
通常电商网站的浏览深度在 3-8 之间属于正常范围。

***
将全量抽样数据的每日 PV/UV/ 浏览深度统计结果，永久保存到一个新的结构化表中。
```sql
CREATE TABLE pv_uv_puv (
    dates CHAR(10),     
    -- 日期字段，存储格式如 '2017-11-25'
    pv INT(9),          
    -- 页面浏览量，最大支持9位数字（足够存亿级数据）
    uv INT(9),           
    -- 独立访客数，最大支持9位数字
    puv DECIMAL(10,1)    
    -- 浏览深度（PV/UV），10位总长度，其中1位小数
);
```

```sql
INSERT INTO pv_uv_puv
SELECT 
    dates,                             
    COUNT(*) AS 'pv',                   
    COUNT(DISTINCT user_id) AS 'uv',   
    ROUND(COUNT(*)/COUNT(DISTINCT user_id),1) AS 'puv'  
FROM user_behavior_sampled
WHERE behavior_type='pv'              
GROUP BY dates;                        
```

## 留存情况
对 temp_behavior 表中的数据按「用户 + 日期」进行去重
```sql
SELECT user_id,dates  -- 要查询的字段：用户ID、日期
FROM temp_behavior    -- 数据源：临时表
GROUP BY user_id,dates;  -- 按「用户ID + 日期」双列分组
```

**基础的自关联（Self-Join）查询**  
```sql
-- 自关联
select * from
(select user_id,dates
from temp_behavior
group by user_id,dates
)a
,(select user_id,dates
from temp_behavior
group by user_id,dates
)b
where a.user_id = b.user_id 
-- 条件1：只匹配同一个用户
AND a.dates < b.dates;
-- 条件2：只保留a的日期早于b的日期的组合
```
- 将「去重后的用户 - 日期活跃记录」作为两个独立的表 a 和 b，通过 user_id 进行自连接，生成同一个用户所有活跃日期的两两组合列表。

- 子查询 a 和 b：完全相同的活跃用户 - 日期去重表  
- 自关联条件：`where a.user_id = b.user_id`  
逻辑：只保留 a 表和 b 表中 user_id 相同的记录，即同一个用户的活跃日期自己和自己连接。

**用户多日留存数计算查询**  
- 核心作用是：统计每个 “初始活跃日期” 的用户，在当日、次日、第 3 天的留存用户数量，是分析用户粘性的核心指标。

留存统计逻辑
```sql
SELECT a.dates
,COUNT(IF(DATEDIFF(b.dates,a.dates)=0,b.user_id,NULL)) retention_0  -- 当日留存
,COUNT(IF(DATEDIFF(b.dates,a.dates)=1,b.user_id,NULL)) retention_1  -- 次日留存
,COUNT(IF(DATEDIFF(b.dates,a.dates)=3,b.user_id,NULL)) retention_3  -- 3日留存
```
（1）`DATEDIFF(b.dates, a.dates)`：计算日期间隔  
（2）`IF(日期间隔=X, b.user_id, NULL)`：筛选符合条件的用户。
- 逻辑：如果日期间隔等于目标值（0/1/3），返回 user_id；否则返回 NULL。  

（3）`COUNT(IF(...)`：统计留存用户数
作用：只统计 IF 函数返回的非 NULL 的 user_id，即 “符合日期间隔条件的用户”。  
- 关键：因为子查询 a 和 b 已经去重，所以同一个用户不会被重复统计。

**计算次日留存率**
- 核心作用是：统计每个 “初始活跃日期” 的用户中，第二天还活跃的用户占比，是衡量用户粘性最核心的指标之一。
```sql
-- 留存率
SELECT a.dates
,COUNT(IF(DATEDIFF(b.dates,a.dates)=1,b.user_id,NULL))-- 分子：次日留存用户数
/COUNT(IF(DATEDIFF(b.dates,a.dates)=0,b.user_id,NULL)) retention_1  -- 分母：当日活跃用户数
FROM
(SELECT user_id,dates
FROM temp_behavior
GROUP BY user_id,dates
)a
,(SELECT user_id,dates
FROM temp_behavior
GROUP BY user_id,dates
)b
WHERE a.user_id = b.user_id AND a.dates <= b.dates
GROUP BY a.dates;
```

留存率完整代码：
```sql
create table retention_rate(
dates char(10),
retention_1 float
);

insert into retention_rate
SELECT a.dates
,COUNT(IF(DATEDIFF(b.dates,a.dates)=1,b.user_id,NULL))/ COUNT(IF(DATEDIFF(b.dates,a.dates)=0,b.user_id,NULL)) retention_1 
FROM
(SELECT user_id,dates
FROM user_behavior_sampled
GROUP BY user_id,dates
)a
,(SELECT user_id,dates
FROM user_behavior_sampled
GROUP BY user_id,dates
)b
WHERE a.user_id = b.user_id AND a.dates <= b.dates
GROUP BY a.dates;
```

## 时间序列分析（一部分）
创建一张表，把「每天 + 每个小时」的浏览、加购、收藏、购买数量存入。

```sql
CREATE TABLE date_hour_behavior(
```

---

总结与洞察
1.  **流量规律**：用户行为具有极强的周期性，晚间 20:00-23:00 是流量高峰。
2.  **转化规律**：流量高峰并不完全等于转化高峰。深夜时段（23:00）虽然流量不是最高，但购买转化率最高。
3.  **运营建议**：
    *   可以在晚间高峰时段加大推广力度获取流量。
    *   针对深夜高转化人群，可以推送高客单价或决策成本较高的商品。
    *   针对白天高流量低转化时段，可以推送促销、秒杀活动来刺激转化。

### 1. 报告概述

本报告基于淘宝平台2017年双十二预热期的用户行为数据，通过对用户浏览、加购、收藏、购买四类核心行为的多维度分析，揭示用户行为规律与转化特征，为大促期间的运营策略制定、资源投放优化提供数据支撑。

- 有效数据量：约800万条记录
- 覆盖用户数：约54万独立用户

### 2. 每日整体行为趋势分析（图表A）

### 4. 日期×小时多维交叉分析（图表C）
4.1 核心发现
1.  **用户行为习惯极其稳定**
    无论工作日还是周末，用户的每日活跃曲线高度一致——每天都有上午10点、下午15点两个小高峰，以及晚上21点一个大高峰。

2.  **预热效果逐日累积**
    从11月25日到12月2日，每天的21点峰值持续攀升，尤其是12月2日的21点，达到整个周期的最高点，说明预热活动的影响力在逐步扩大。

3.  **周末效应明显**
    12月2日（周六）和12月3日（周日）的全时段流量均显著高于工作日，符合电商平台“周末流量更高”的普遍规律。

4.2 业务意义
可以基于这个稳定的规律，提前规划不同时段的运营活动，最大化流量利用效率。

### 5. 24小时转化率深度分析（图表D）
5.1 核心发现
1.  **购买转化率（绿色线）：呈现单峰分布**
    - 高峰：上午10点（转化率0.26，全天最高）
    - 低谷：早上5-6点（转化率0.1281，全天最低）
    - 次高峰：下午15点

2.  **加购收藏率（蓝色线）：整体相对平缓**
    - 高峰：晚上22-23点（转化率0.96，全天最高）
    - 低谷：下午18点

3.  **关键反直觉规律**
    加购收藏转化率达到峰值时，购买转化率反而不高；反之，购买转化率最高的时段，加购收藏率并非最高。

5.2 业务意义
不同时段的用户群体和行为目的存在显著差异，不能简单地“哪里流量多就往哪里投”，而应该根据不同时段的转化率特征，制定差异化的运营策略。

### 6. 核心洞察总结
1.  **预热策略成功**：提前3天启动预热，大促前1天达到流量峰值，转化与流量同步增长
2.  **用户习惯稳定**：每日活跃曲线高度一致，晚上21点是全天流量高峰
3.  **时段价值分化**：
    - 上午10点：**转化黄金时段**（流量中等，购买转化率全天最高）
    - 晚上21-23点：**蓄水黄金时段**（流量全天最高，加购收藏率全天最高）
    - 下午15点：**次高峰转化时段**（流量次高，转化表现良好）
    - 凌晨0-4点：**精准用户时段**（流量低，但用户购买意愿强）

### 7. 可落地业务建议
7.1 大促预热期整体运营策略
| 核心目标 | 具体策略 | 落地细节 |
|----------|----------|----------|
| 流量增长 | 延续阶梯式预热节奏 | 提前3天启动预热，通过红包、优惠券、活动预告逐步提升用户关注度，在大促前1天（周六）达到流量峰值 |
| 转化稳定 | 优化商品详情页引导 | 保留当前加购/收藏按钮的展示逻辑，强化“加购享优惠”“收藏优先发货”等权益提示 |
| 周末放大 | 周末加码运营资源 | 12月2-3日周末时段，增加首页活动曝光位、延长核心直播间时长，利用周末流量高峰放大预热效果 |

7.2 分时段精细化运营策略
（1）上午10点：转化黄金时段
- **核心目标**：直接收割转化，提升订单量
- **推荐动作**：
  - 推送限时秒杀、小额无门槛优惠券、订单催付提醒，刺激用户即时下单
  - 针对已加购/收藏用户，推送“限时降价”“库存紧张”消息，减少用户犹豫时间
  - 安排核心客服团队在线，快速解答商品规格、物流、售后等问题

（2）晚上21-23点：蓄水黄金时段
- **核心目标**：引导用户加购收藏，为大促蓄水
- **推荐动作**：
  - 重点推送爆款商品推荐、“收藏有礼”“加购享立减”活动，降低用户加购门槛
  - 强化商品种草内容，优先展示优质买家秀、达人测评、直播讲解
  - 推送“大促价提前锁定”“加购用户专属权益”等信息，引导用户提前锁定商品

（3）下午15点：次高峰转化时段
- **核心目标**：轻度转化+用户唤醒
- **推荐动作**：
  - 推送复购率高的日用品推荐、会员专属折扣权益
  - 针对7天内有浏览记录但未下单的用户，推送“浏览商品降价提醒”

（4）凌晨0-4点：精准用户运营时段
- **核心目标**：服务高意向用户，提升转化效率
- **推荐动作**：
  - 安排1-2名资深客服值班，及时响应夜间用户咨询
  - 推出“深夜专属95折”“限时免邮”等夜间专属福利

7.3 广告投放优化策略
| 时段 | 投放目标 | 预算占比 | 核心动作 |
|------|----------|----------|----------|
| 晚上20-22点 | 曝光+加购收藏 | 60% | 投放预热活动广告、爆款商品广告，最大化流量曝光与加购行为 |

7.4 客服与供应链优化策略
- **客服排班**：
  - 晚上20-22点：安排80%的客服人员值班，将平均响应时间控制在30秒以内
  - 上午10点：安排60%的客服人员值班，重点处理下单、支付相关问题

  - 提前备好TOP100爆款商品的库存，确保库存充足

7.5 后续优化方向

2.  持续追踪预热期加购收藏用户的后续转化路径，优化蓄水用户的触达时机与内容

### 8. 附录：指标定义与数据说明
8.1 核心指标定义
- **PV（页面浏览量）**：用户每打开一次商品页面计为1次PV
- **UV（独立访客数）**：同一天内同一个用户只计为1次UV
- **加购率**：加购行为数 / PV
- **收藏率**：收藏行为数 / PV
- **购买转化率**：购买行为数 / PV

8.2 数据处理说明
1.  数据清洗：剔除了时间范围外的脏数据和重复记录
2.  抽样方式：采用按用户ID分层随机抽样，确保同一个用户的所有行为记录被同时保留
3.  统计口径：所有指标均基于抽样数据计算，未进行全量还原

## 用户转化率分析
从「用户角度」统计行为用户数
```sql
-- 统计各类行为用户数（从用户角度分析）
select behavior_type
,count(distinct user_id) user_num
from temp_behavior
group by behavior_type
order by behavior_type desc;
```
- `COUNT(DISTINCT user_id)` —— 对 `user_id` 去重后计数
- 统计有多少个不同的用户产生过该行为

存储统计结果
```sql
-- 存储
create table behavior_user_num(

insert into behavior_user_num
select behavior_type
,count(distinct user_id) user_num
from user_behavior_sampled
group by behavior_type
order by behavior_type desc;

select * from behavior_user_num;
```
- 将临时表的验证逻辑，应用到全量抽样表 `user_behavior_sampled` 上，并将结果永久保存到新表 `behavior_user_num` 中。

计算用户层面的“购买/浏览”比例
```sql
select 53865/78799 -- buy/pv=0.6836
```

- 业务含义：在所有浏览过商品的用户中，有68.36%的用户最终完成了购买

从「行为角度」统计行为总次数

```sql
-- 统计各类行为的数量（从行为角度分析）

FROM temp_behavior
GROUP BY behavior_type
ORDER BY behavior_type DESC;
```
- `COUNT(*)` —— 直接统计记录总数，不去重
- **业务含义**：统计“**该行为总共发生了多少次**”

## 行为路径分析

```
原始行为数据 → 按用户+商品聚合行为次数 → 标准化为0/1（是否做过） 
→ 筛选购买用户 → 拼接4位路径编码 → 关联翻译表 → 统计各路径数量
```

---
 1. 创建结果存储表
```sql
CREATE TABLE path_result(
path_type CHAR(4),        -- 存储4位路径编码（如 '1001'）
DESCRIPTION VARCHAR(40),  -- 存储路径的中文描述（如 '浏览后购买了'）
num INT                   -- 存储该路径的用户数量
);
```
作用：创建一张空表，用于最终存储分析结果。

---

 2. 视图1：按「用户+商品」聚合行为次数
```sql
CREATE VIEW user_behavior_view AS
SELECT user_id,item_id
,COUNT(IF(behavior_type='pv',behavior_type,NULL)) 'pv'    -- 浏览次数
,COUNT(IF(behavior_type='fav',behavior_type,NULL)) 'fav'  -- 收藏次数
,COUNT(IF(behavior_type='cart',behavior_type,NULL)) 'cart'-- 加购次数
,COUNT(IF(behavior_type='buy',behavior_type,NULL)) 'buy'  -- 购买次数
FROM user_behavior_sampled
GROUP BY user_id,item_id;
```
- 按 `user_id`（用户）和 `item_id`（商品）分组，统计**每个用户对每个商品**的四类行为分别发生了多少次。

---

3. 视图2：行为标准化（0/1编码）
```sql
CREATE VIEW user_behavior_standard AS
SELECT user_id
,item_id
,(CASE WHEN pv>0 THEN 1 ELSE 0 END) 浏览了    -- 只要浏览过≥1次，记为1，否则0
,(CASE WHEN fav>0 THEN 1 ELSE 0 END) 收藏了    -- 只要收藏过≥1次，记为1，否则0
,(CASE WHEN cart>0 THEN 1 ELSE 0 END) 加购了    -- 只要加购过≥1次，记为1，否则0
,(CASE WHEN buy>0 THEN 1 ELSE 0 END) 购买了    -- 只要购买过≥1次，记为1，否则0
FROM user_behavior_view;
```
- 用 `CASE WHEN` 将“次数”简化为“是否做过”（0或1）。
- 路径分析只关心用户有没有浏览/收藏/加购，不关心浏览了几次，这样可以将复杂的行为简化为4个二进制位。

---

4. 视图3：生成购买路径编码
```sql
CREATE VIEW user_behavior_path AS
SELECT *,
CONCAT(浏览了,收藏了,加购了,购买了) 购买路径类型  -- 拼接4个0/1，形成4位字符串
FROM user_behavior_standard AS a
WHERE a.购买了>0;  -- 只筛选有购买行为的记录（没购买的不需要分析路径）
```
  1. `CONCAT(浏览了,收藏了,加购了,购买了)`：将4个0/1字段拼接成一个4位字符串。
  2. `WHERE a.购买了>0`：只保留最终产生了购买的记录，因为我们只关心“购买用户的转化路径”。

---

5. 视图4：统计各路径的数量
```sql
CREATE VIEW path_count AS
SELECT 购买路径类型
,COUNT(*) 数量  -- 统计每种路径有多少个用户
FROM user_behavior_path
GROUP BY 购买路径类型
ORDER BY 数量 DESC;  -- 按数量从多到少排序
```
- 统计每种4位路径编码对应的用户数量，找出最常见的购买路径。

---

 6. 创建“路径翻译表”（将编码转化为业务语言）
```sql
CREATE TABLE renhua(
path_type CHAR(4),        -- 4位路径编码
DESCRIPTION  VARCHAR(40)  -- 对应的中文描述
);

INSERT INTO renhua
VALUES('0001','直接购买了'),
('1001','浏览后购买了'),
('0011','加购后购买了'),
('1011','浏览加购后购买了'),
('0101','收藏后购买了'),
('1101','浏览收藏后购买了'),
('0111','收藏加购后购买了'),
('1111','浏览收藏加购后购买了');
```
- 建立“编码”到“自然语言”的映射关系，让分析结果可读。

---

7. 关联翻译表并插入最终结果
```sql
INSERT INTO path_result
SELECT path_type,DESCRIPTION,数量 FROM 
path_count p
JOIN renhua r
ON p.购买路径类型=r.path_type
ORDER BY 数量 DESC;

-- 最终查询展示
SELECT * FROM path_count p
JOIN renhua r
ON p.购买路径类型=r.path_type
ORDER BY 数量 DESC;
```
- **逻辑**：将统计结果 `path_count` 和翻译表 `renhua` 关联，把4位编码替换成易懂的中文描述。
- 最终结果示例：
  | path_type | DESCRIPTION          | 数量  |
  |-----------|----------------------|-------|
  | 1001      | 浏览后购买了         | 45000 |
  | 1011      | 浏览加购后购买了     | 32000 |
  | 0001      | 直接购买了           | 18000 |

---

业务价值总结
1.  **识别核心转化路径**：知道用户最常通过哪种路径购买（是“浏览直接买”还是“浏览→加购→买”）。
2.  **优化产品流程**：
    - 如果“浏览后购买了”占比最高：说明用户决策很快，要优化商品详情页的“立即购买”按钮。
    - 如果“浏览加购后购买了”占比最高：说明用户喜欢加购对比，要优化“购物车”页面的催付提醒。
3.  **发现异常路径**：如果“直接购买了”占比高，可能是复购用户多，可以针对性做复购运营。

## 基于简化 RFM 模型思想的用户价值分层实现
第一步：计算用户核心指标
```sql
-- 创建RFM结果存储表
DROP TABLE IF EXISTS rfm_model;
CREATE TABLE rfm_model(
user_id INT,
frequency INT,  
recent CHAR(10) 
);

-- 计算每个购买用户的F值和R值
INSERT INTO rfm_model
SELECT user_id
,COUNT(user_id) '购买次数'  -- F值：用户在9天内的总购买次数
,MAX(dates) '最近购买时间'  -- R值：用户最后一次购买的日期
FROM user_behavior_sampled
WHERE behavior_type='buy'  -- 只统计有购买行为的用户
GROUP BY user_id
ORDER BY 2 DESC,3 DESC;
```
- 业务含义：
  - F值越高：用户购买越频繁，忠诚度越高
  - R值越新（日期越接近12月3日）：用户越活跃，流失风险越低

---
第二步：指标标准化打分（为分类做准备）
将原始的“购买次数”和“购买日期”转化为**1-5分的统一评分**，消除不同指标的量纲差异，方便后续分层。
 1. F值（购买频率）打分
```sql
ALTER TABLE rfm_model ADD COLUMN fscore INT;

UPDATE rfm_model
SET fscore=CASE
WHEN frequency BETWEEN 100 AND 262 THEN 5 
WHEN frequency BETWEEN 50 AND 99 THEN 4
WHEN frequency BETWEEN 20 AND 49 THEN 3
WHEN frequency BETWEEN 5 AND 20 THEN 2
ELSE 1  
END;
```
 2. R值（最近购买时间）打分
```sql
ALTER TABLE rfm_model ADD COLUMN rscore INT;

UPDATE rfm_model
SET rscore=CASE
WHEN recent = '2017-12-03' THEN 5  
WHEN recent IN ('2017-12-01','2017-12-02') THEN 4
WHEN recent IN ('2017-11-29','2017-11-30') THEN 3
WHEN recent IN ('2017-11-27','2017-11-28') THEN 2
ELSE 1  
END;
```
---
第三步：用户分层

1. 计算平均分作为分层阈值
```sql
SET @f_avg=NULL;
SET @r_avg=NULL;
SELECT AVG(fscore) INTO @f_avg FROM rfm_model;  -- 所有用户的F值平均分
SELECT AVG(rscore) INTO @r_avg FROM rfm_model;  -- 所有用户的R值平均分
```
- 采用**平均分二分法**：将用户分为“高于平均”和“低于平均”两类，是最常用、最简单有效的分层方法。

2. 四象限用户分类
```sql
ALTER TABLE rfm_model ADD COLUMN class VARCHAR(40);
UPDATE rfm_model
SET class = CASE
WHEN fscore>@f_avg AND rscore>@r_avg THEN '价值用户'  -- 第一象限：F高+R高
WHEN fscore>@f_avg AND rscore<@r_avg THEN '保持用户'  -- 第二象限：F高+R低
WHEN fscore<@f_avg AND rscore>@r_avg THEN '发展用户'  -- 第四象限：F低+R高
WHEN fscore<@f_avg AND rscore<@r_avg THEN '挽留用户'  -- 第三象限：F低+R低
END;
```
- 坐标轴对应：
  - 横轴：R值（越往右，R值越高，用户越活跃）
  - 纵轴：F值（越往上，F值越高，用户越忠诚）

---
第四步：统计分层结果
```sql
-- 统计各分区用户数
SELECT class,COUNT(user_id) FROM rfm_model
GROUP BY class;
```
---
模型简化说明：由于原数据集没有消费金额（M值），因此采用简化的RF模型。如果后续有金额数据，只需增加M值的计算和打分，即可升级为完整的RFM模型。

## 商品按热度分类


整体逻辑概览
嵌套子查询 + 窗口函数：
1.  内层：先统计“每个品类下每个商品”的浏览量。
2.  窗口函数：在每个品类内部，按浏览量给商品排名。
3.  筛选：只保留每个品类的第1名。
4.  外层：在所有品类的爆款中，再取浏览量最高的前10个，存入结果表。

---
 1. 最内层：统计“品类+商品”的浏览量
```sql
select category_id,item_id
,COUNT(IF(behavior_type='pv',behavior_type,NULL)) '品类商品浏览量'
FROM user_behavior_sampled
GROUP BY category_id,item_id
```
 2. 窗口函数 `RANK()` 进行品类内排名
```sql
rank() over(partition by category_id order by COUNT(IF(behavior_type='pv',behavior_type,NULL)) desc) r
```
（1）`partition by category_id`（按品类分组）
- 作用：将数据**按品类切分成独立的小组**。
- 效果：品类123的商品在一起排名，品类456的商品在一起排名，互不干扰。

（2）`order by ... desc`（按浏览量降序）
- 作用：在每个品类内部，按浏览量从高到低排序。

（3）`rank()`（排名函数）
- 作用：给每个商品生成一个排名数字（1, 2, 3...）。
---

 3. 筛选：只保留每个品类的第1名
```sql
where a.r=1
```
- 作用：只保留排名 `r=1` 的记录，即每个品类下浏览量最高的那个商品。

---

4. 外层：最终排序并取前10
```sql
order by a.品类商品浏览量 desc
limit 10
```
- 作用：在所有品类爆款中，再按浏览量从高到低排序，取前10名。

## 商品转化率分析
统计每个商品/品类的“浏览、收藏、加购、购买”行为次数，并计算基于用户数的真实转化率。



商品转化率分析

1. 创建商品明细结果表
```sql
CREATE TABLE item_detail(
item_id INT,          -- 商品ID
pv INT,               -- 浏览行为次数
fav INT,              -- 收藏行为次数
cart INT,             -- 加购行为次数
buy INT,              -- 购买行为次数
user_buy_rate FLOAT   -- 商品用户转化率
);
```
作用：创建一张空表，用于存储每个商品的全链路行为数据和最终转化率。

---

 2. 插入并计算商品数据
```sql
INSERT INTO item_detail
SELECT item_id
,COUNT(IF(behavior_type='pv',behavior_type,NULL)) 'pv'    -- 统计该商品的浏览总次数
,COUNT(IF(behavior_type='fav',behavior_type,NULL)) 'fav'  -- 统计该商品的收藏总次数
,COUNT(IF(behavior_type='cart',behavior_type,NULL)) 'cart'-- 统计该商品的加购总次数
,COUNT(IF(behavior_type='buy',behavior_type,NULL)) 'buy'  -- 统计该商品的购买总次数
-- 【核心指标】商品用户转化率
,COUNT(DISTINCT IF(behavior_type='buy',user_id,NULL))/COUNT(DISTINCT user_id) 商品转化率
FROM user_behavior_sampled
GROUP BY item_id
ORDER BY 商品转化率 DESC;
```


2.  商品用户转化率：
    ```sql
    COUNT(DISTINCT IF(behavior_type='buy',user_id,NULL))  -- 分子：购买过该商品的独立用户数
    / COUNT(DISTINCT user_id)  -- 分母：接触过该商品的所有独立用户数
    ```
    - 业务含义：每100个接触过该商品的用户中，有多少人最终购买了它

---

品类转化率分析


```sql
-- 创建品类明细结果表

```
